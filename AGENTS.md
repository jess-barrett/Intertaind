This version has breaking changes — APIs, conventions, and file structure may all differ from your training data. Read the relevant guide in `node_modules/next/dist/docs/` before writing any code. Heed deprecation notices.

## Monorepo layout

- `apps/web` — Next.js 16 web app
- `apps/mobile` — Expo (React Native) mobile app, SDK 56
- `packages/types` — hand-curated domain types (`@intertaind/types`)
- `packages/media` — shared external-API types + pure normalization (`@intertaind/media`)
- `packages/design-system` — canonical design tokens (`@intertaind/design-system`)
- `packages/supabase` — generated database types (`@intertaind/supabase`)
- `supabase/` — shared database migrations (and, later, Edge Functions)

pnpm workspace + Turborepo. Run apps from the root: `pnpm dev:web`, `pnpm dev:mobile`.

| Command | What it does |
|---|---|

## Critical conventions

### Server-only vs public secrets
- **`apps/web/.env.local`** carries Supabase keys + every external API secret (TMDB, IGDB/Twitch, Google Books, Hardcover). These run only in Server Components / Server Actions / API routes — they never reach the browser.
- **`apps/mobile/.env`** carries ONLY `EXPO_PUBLIC_SUPABASE_URL` and `EXPO_PUBLIC_SUPABASE_ANON_KEY`. Anything you add here is shipped in the mobile bundle to users' devices. **Server secrets MUST NOT go here.** External-API calls that need server secrets migrate to Supabase Edge Functions later (deferred Phase 4 in `docs/plans/2026-06-11-monorepo-and-mobile-scaffold.md`).
- The root `/.env.local` is gone. Anything at the repo root is unread by both apps.

### Design tokens have a single source of truth
- Edit `packages/design-system/src/tokens.cjs` first.
- Mirror the change into `apps/web/src/app/globals.css` (the `:root` block).
- `pnpm --filter @intertaind/design-system test` parses both and fails CI on drift.
- See `packages/design-system/AGENTS.md` for the workflow.

### Generated DB types are read-only
- `packages/supabase/src/database.types.ts` is generated by `pnpm gen:types`. Never hand-edit.
- After every migration: `pnpm gen:types` and commit the regenerated file with the migration.
- See `packages/supabase/AGENTS.md`.

### Server actions auto-revalidate
- In Next.js 16, calling a Server Action from a client component already revalidates the calling route's Router Cache. Adding an explicit `router.refresh()` after the await stacks a redundant RSC fetch (×2 with React StrictMode in dev — the source of multi-burst refresh storms).
- Default: **don't** add `router.refresh()` after a server action. Only add one when you need to refresh data that the action didn't touch (cross-route invalidation), and prefer `revalidatePath` server-side over `router.refresh` client-side.

## Engineering principles

- **Quality over speed.** Solo-dev side project aiming at industry-grade architecture for scale. Don't reach for the quickest patch when a principled fix is available; see [the principled-fixes memory note for context on past corrections](../../Users/Jess/.claude/projects/c--Intertaind-intertaind/memory/feedback_principled_fixes.md).
- **Plan substantive work.** For anything multi-file or architectural, write a plan first (under `docs/plans/YYYY-MM-DD-name.md`) before making structural changes. Plans there serve as reading material for future agents.
- **Run the build.** After any structural change, `pnpm typecheck` and (for completed work) the full `pnpm ci`. CI lights are the contract.
- **Verify cross-platform changes both ways.** Anything in `packages/*` affects web AND mobile. Confirm web builds AND mobile typechecks.
