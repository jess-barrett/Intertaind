# apps/mobile — Expo + React Native conventions

Read the root `AGENTS.md` first. Notes here are mobile-specific only.

## Expo HAS CHANGED

This is Expo SDK 56 + React Native 0.85 + React 19.2. Read the exact versioned docs at https://docs.expo.dev/versions/v56.0.0/ before writing code that touches an Expo or RN surface. Training data is often a major version behind.

## Dev environment (Windows)

- **Android emulator only** on Windows. iOS requires macOS. For iOS verification: cloud Mac or a borrowed machine, weekly cadence is fine.
- **Dev client, not Expo Go.** The project is committed to `expo-dev-client` (see `48ccccb` commit). Expo Go will not work.
- **JDK 17 required.** Gradle / React Native do NOT support JDK 20+. Microsoft OpenJDK 17 at `C:\Program Files\Microsoft\jdk-17.0.x.x-hotspot` with `JAVA_HOME` pointed at it. Android Studio's bundled JBR is JDK 21 — incompatible (triggers `IBM_SEMERU` toolchain errors).
- AEHD (Android Emulator Hypervisor Driver) provides hardware acceleration on Windows 10 Home. Requires VT-x enabled in BIOS.

## Daily dev loop

1. Boot the AVD: Android Studio → Device Manager → ▶ on `Pixel7_API34`.
2. From repo root: `pnpm --filter mobile dev` (or `pnpm dev:mobile` — both work; the bare `dev:mobile` goes through Turbo which can swallow Expo's keypress prompt, prefer `--filter mobile dev` if pressing keys in Metro).
3. Dev client on the emulator auto-connects to Metro.
4. JS changes hot-reload. Native changes (Expo plugin add, `app.json` native-config change, new native module) require `expo run:android` to recompile the dev client APK.

The native `android/` folder is generated by prebuild and gitignored — never commit it, never hand-edit it. Configure native bits via `app.json` and config plugins instead.

> **⚠️ After `expo install <pkg>` in this pnpm workspace, run `pnpm install` at the repo root, then restart Metro with `--clear`.** `expo install` updates `package.json` + `pnpm-lock.yaml` and can leave the hoisted `node_modules` half-linked — `tsc` and `node`'s resolver still find the package (via the root), but Metro fails with `Unable to resolve module <pkg>` for it while resolving its siblings fine. `pnpm install` reconciles the workspace linking (it's a no-op if already consistent). This bit us adding `@gorhom/bottom-sheet` (2026-07). Whether a full `expo run:ios` rebuild is *also* needed depends on the pod: a JS-only lib (`@gorhom/bottom-sheet`) needs only `pnpm install` + Metro `--clear`; a new NATIVE module (its own pod) additionally needs a dev-client rebuild.

## Architecture

- **Tailwind via NativeWind 4.x**, config in `tailwind.config.js`. Colors come from `@intertaind/design-system/tokens.cjs` — do not hand-add colors to the tailwind theme. The content glob includes `packages/types/src/**/*` so shared class strings (e.g. `MEDIA_TYPE_CONFIG`) compile.
- **Routing via Expo Router** — file-based, the analogue of Next's App Router. Root layout is `src/app/_layout.tsx`.
- **Tab shell = per-tab Stacks + a SHARED detail route (expo-router "shared routes", SDK-56 doc-blessed).** The tab screens live in an ARRAY-GROUP folder `src/app/(tabs)/(index,explore)/`, which expo-router extrapolates at build time into two real per-tab Stacks — `(index)` (Trending, anchor `index.tsx`) and `(explore)` (anchor `explore.tsx`) — each with its own `_layout.tsx` Stack and its own back stack. `media/[id].tsx` sits in that same folder, so it's shared INTO both stacks and `router.push("/media/<id>")` pushes it onto the CURRENT tab's stack. Result: the custom bottom navbar (rendered by `(tabs)/_layout.tsx`, ABOVE the Stacks) stays visible on detail, native/gesture back returns within the tab, and switching tabs preserves each tab's stack. Consequences to remember: (1) the Tabs navigator's route names — and the custom bar's `ROUTE_ICONS` keys in `components/nav/bottom-tab-bar.tsx` — are the GROUP names `(index)`/`(explore)` (parentheses kept), not `index`/`explore`; (2) anchors auto-resolve because each tab screen is named after its group (so NO `unstable_settings` needed — keep it that way if you add a tab: name the screen to match the group, or add a `unstable_settings` anchor); (3) the Trending home href is `/` (route `(tabs)/(index)`); `/(tabs)` is no longer a leaf href. When adding a NEW tab, add its group to the array folder name AND a matching `Tabs.Screen`/`ROUTE_ICONS` entry.
- **App-wide providers** are hoisted into `src/components/providers.tsx`. Add new React contexts (auth, theme, etc.) there in order of dependency. The root layout just renders `<Providers>` around the tree.
- **Supabase client in `src/lib/supabase.ts`** — see Auth section below for session storage.

## Data layer — TanStack Query

**ALL** data fetching goes through `src/queries/`. No inline `useEffect` + `supabase.from(...)` in screens. The Trending screen (`src/app/(tabs)/(index,explore)/index.tsx`) is the canonical example — copy that shape.

Structure:
- `src/queries/keys.ts` — the query-key factory. Hierarchical, type-safe, single source of truth. Never inline `["media", "trending"]` in a hook — always `queryKeys.media.trending()`.
- `src/queries/<resource>.ts` — one file per resource (`media.ts`, `userShelves.ts`, etc.). Each file exports hooks: `useTrendingMedia()`, `useMediaDetail(id)`, etc.

Conventions:
- Hook names: `use<Resource><View>` (`useTrendingMedia`, `useUserShelves`).
- Mutations: `use<Resource><Action>Mutation` (`useTrackMediaMutation`, `useToggleFavoriteMutation`). On success, invalidate the hierarchical key at the right level — e.g. a `useTrackMediaMutation` invalidates `queryKeys.user.shelves(userId)` AND `queryKeys.media.detail(mediaId)`.
- Types: `Pick<Tables<'foo'>, ...>` from `@intertaind/supabase` documents exactly which columns the query reads. Don't `select('*')` — denormalize only what the caller renders. Saves bandwidth and makes column-level access audits trivial.
- Errors: throw inside `queryFn` on Supabase errors. TanStack surfaces them via `error` for the component to render.

QueryClient defaults live in `src/components/providers.tsx`: 5-min stale, 30-min gc, retry once except on 4xx (real bugs, don't retry). Don't override per-query unless there's a documented reason.

## Auth architecture

Email/password + (planned) Google/Apple. The pieces, outer to inner:

- **`src/components/auth-provider.tsx`** (`AuthProvider` + `useAuth()`) — the single source of "who is signed in." Subscribes to `supabase.auth.getSession()` (initial) + `onAuthStateChange` (updates) and exposes `{ session, user, profileStatus, loading, refreshProfileStatus }`. `profileStatus` is 4-state: `"none"` (signed out) · `"missing"` (signed in, no `profiles` row → needs username) · `"present"` · `"error"` (transient profile-fetch failure — distinct so gating doesn't strand the user). A monotonic `syncSeq` ref guards against out-of-order profile fetches; `loading` is cleared in a `finally` so a failed session read can't strand the splash. Lives inside `QueryClientProvider` (in `providers.tsx`).
- **Route groups + gating** — `src/app/(auth)/` (login, signup, setup-username) and `src/app/(tabs)/` (the app). Root `src/app/_layout.tsx`'s `RootNavigator` is the ONLY place that redirects: `loading`→splash; `!session`→login; `missing`→setup-username; `present`→tabs; `error`→retry view. **Screens must never call `router.push/replace` for auth transitions** — mutate state and let gating react. (Considering `Stack.Protected` later; see the plan's deferred items.)
- **`src/queries/auth.ts`** — all auth mutations (`useSignIn/SignUp/SignOut/CreateProfileMutation`), per the TanStack convention. Email/password signup passes `username` in `options.data` so the `handle_new_user` DB trigger creates the profile; OAuth/Apple users have no profile → setup-username → `useCreateProfileMutation` inserts it client-side under the `profiles_insert_self` RLS policy (mirrors web's `createInitialProfile`). Username rules come from the shared `validateUsername` in `@intertaind/types` — do not re-inline a regex.

> **⚠️ NEVER call Supabase methods inside `onAuthStateChange`.** The client uses `lock: processLock`, and `auth-js` invokes the `onAuthStateChange` callback **while holding the auth lock** (e.g. `_emitInitialSession` → `await _useSession(cb)`). Any `supabase.from(...)` / `supabase.auth.*` call inside the callback re-acquires that lock and **deadlocks on cold start** — the concurrent `getSession()` never settles, so `loading` never clears (infinite splash). Keep the callback **synchronous** and defer any Supabase work with `setTimeout(() => …, 0)` so it runs after the lock releases. This bit us once (July 2026); the fix + rationale is commented in `auth-provider.tsx`.

## Session storage

The Supabase client in `src/lib/supabase.ts` persists sessions to **`expo-secure-store`** via the chunked adapter in `src/lib/secure-storage.ts`. SecureStore has a 2KB per-value limit on Android; Supabase session tokens often exceed this, so the adapter splits values transparently across `<key>.0`, `<key>.1`, … entries. The adapter also detects when the native module is absent (Node typed-route generation / web bundle) via a call-time latch and falls back to a non-persistent in-memory store, so evaluating the module graph off-device can't crash.

Do NOT switch session storage back to `AsyncStorage` — it stores tokens in plaintext on disk and any other app on a rooted device can read them.

## Scrollable screens must clear the navbar

The custom bottom navbar (`components/nav/bottom-tab-bar.tsx`) is rendered by the `Tabs` navigator ABOVE the per-tab Stacks, so it OVERLAYS the bottom of every screen. A `ScrollView`/`FlatList` whose content can reach the bottom edge will hide its last rows behind the bar unless it reserves that space.

**Convention:** every scrollable screen pads its bottom by `useBottomInset()` (`src/lib/use-bottom-inset.ts`) — a hook returning the points to reserve = the bar's content height + `Math.max(safe-area bottom, 8)`. Apply it via the scroll container's `contentContainerStyle` (NOT the outer `className`, which doesn't reach scroll content):

```tsx
const bottomInset = useBottomInset();
// FlatList / ScrollView:
contentContainerStyle={{ paddingBottom: bottomInset }}
// If the screen already has intentional content padding, ADD to it:
contentContainerStyle={{ paddingBottom: 48 + bottomInset }}
```

Non-scrolling screens (e.g. the `explore.tsx` placeholder) don't need it — their content can't scroll under the bar. Applied screens: Trending (`(tabs)/(index,explore)/index.tsx`) and media detail (`(tabs)/(index,explore)/media/[id].tsx`).

Why the hook COMPOSES the inset (bar constants + safe area) instead of react-navigation's `useBottomTabBarHeight()`: that hook returns the DEFAULT bar's measured height, but our custom `tabBar` never reports its layout to the navigator's height-callback context, so the hook falls back to a hardcoded UIKit constant that doesn't match our bar. The bar exports its own geometry constants (`BAR_CONTENT_HEIGHT`, `BAR_MIN_BOTTOM_PADDING`) so the hook stays in lockstep with the bar — change the bar's layout, change those constants. Full rationale is in `use-bottom-inset.ts`.

## Patterns

- Mobile primitives only: `View`, `Text`, `Image`, `FlatList`, etc. No HTML elements, no `<div>`, no `onClick` (use `onPress`). For images, import `Image` from `@/components/image` (NOT directly from `expo-image`) — that module wires NativeWind `className` support once; direct imports silently lack it depending on import order.
- DB types from `@intertaind/supabase`. Domain types from `@intertaind/types`. Same rules as web.
- Don't lift web hover idioms — RN has no hover state on touch. Translate hover-based affordances into press/long-press or visual emphasis.
